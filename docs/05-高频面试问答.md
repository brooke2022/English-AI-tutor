## 一、架构 / 工程类

### Q1：为什么做前后端分离？
**考点**：是否理解前后端分离的本质收益。
**要点**：
- 职责解耦：前端专注 UI/交互，后端专注业务/数据，可以独立开发、独立部署、独立伸缩。
- 后端是纯 API，天然能服务多端（未来加移动端不用重写）。
- 部署上各自容器化：前端是静态资源走 Nginx/CDN，后端是常驻服务，两者资源需求和伸缩策略不同。
- 代价：跨域（CORS）、认证要自己设计（不能靠 session cookie 那套简单方案）、SEO 需额外处理——这个项目是工具型应用，SEO 不是重点。

### Q2：NestJS 的 Module / Controller / Service / Guard 各是什么？
**考点**：对所用框架的核心概念是否清楚。
**要点**：
- **Module**：按业务领域组织代码的单元，声明这个领域用到的 Controller、Service 和依赖。
- **Controller**：处理 HTTP 路由，只负责「收请求、调 Service、返响应」，不写业务逻辑。
- **Service**：业务逻辑的载体，通过依赖注入互相调用（如 `BookingsService` 注入 `EmailService`）。
- **Guard**：在请求到达 Controller 前做拦截判断，项目里 `JwtAuthGuard` 做鉴权、`RolesGuard` 做角色权限。
- 还可以提：**Pipe**（`ValidationPipe` 做入参校验）、**自定义装饰器**（`@Public` / `@Roles` / `@CurrentUser`）。

### Q3：你的项目怎么做权限控制（RBAC）？
**考点**：多角色系统的权限设计。
**要点**：
- 三种角色 STUDENT / TEACHER / ADMIN，存在 `User.role`，也写进 JWT payload。
- **后端**：全局 `JwtAuthGuard` 默认所有接口都要登录，`@Public()` 装饰器标记例外（如登录注册）；`RolesGuard` + `@Roles()` 装饰器声明接口允许的角色。
- **业务层还有归属校验**：比如「接受预约」不仅要 TEACHER 角色，还要校验「是这个预约的外教本人」。
- **前端**：`ProtectedRoute` 组件做路由级拦截——未登录跳登录页、角色不符跳到该角色自己的首页。前端拦截只是体验优化，**真正的安全边界在后端**。

### Q4：CI 做了什么？
**考点**：是否有工程化意识。
**要点**：GitHub Actions 在 push / PR 时并行跑：前端 `lint + build`、后端 `prisma generate + build + test`；main 分支额外跑 Docker 构建 smoke test。作用是「合并前就拦住构建失败和明显错误」。延伸：目前还没做自动部署（CD），是可补的点。

### Q5：Docker 多阶段构建解决什么问题？
**考点**：对容器化的理解。
**要点**：
- 构建需要的东西（Node、全部依赖、源码、编译工具）和运行需要的东西是不一样的。
- 多阶段：第一阶段「构建」装全套依赖、编译；第二阶段「运行」只复制产物。
- 前端最终镜像只有 Nginx + 静态文件，后端只有精简 Node + 编译产物 + 生产依赖——**镜像体积大幅减小，攻击面也更小**。
- 后端运行阶段还用了非 root 用户、启动时自动 `prisma migrate deploy`。

## 二、数据库类

### Q6：怎么保证同一时段不会被重复预约？
**考点**：并发竞态的处理（高频必问）。
**要点**：
- 不能用「先查有没有冲突、再插入」——check 和 act 之间有时间窗，并发下两个请求会都通过检查。
- 做法：`Booking` 表加复合唯一约束 `@@unique([teacherId, slotTime])`，直接 INSERT 让数据库裁决，捕获唯一约束冲突错误（Prisma 的 P2002）转成 409 友好提示。
- 优势：不需要分布式锁、不需要串行化事务，靠数据库本身的约束，可靠且高性能。
- （详见 [04 难点 1](./04-难点与解决方案.md#难点-1--并发下的重复预约竞态)）

### Q7：为什么 refresh token 要哈希后再存数据库？
**考点**：安全意识。
**要点**：
- refresh token 长期有效（30 天），是高价值凭证。
- 如果明文存库，一旦数据库被拖，所有用户的 refresh token 直接泄露、可被冒用。
- 存 `SHA256(token)`：库被拖也拿不到能用的 token；验证时把传入的 token 哈希后比对即可。
- 类比密码不能明文存——区别是 refresh token 是高熵随机串，不需要 bcrypt 那种慢哈希加盐，SHA256 足够。

### Q8：数据库索引怎么设计的？
**考点**：是否有性能意识。
**要点**：举几个真实的：
- `Booking` 上 `[studentId, status]` 和 `[teacherId, status]`——「查我的预约 + 按状态筛」是高频操作。
- `TeacherProfile` 上 `[status]`（列表只查 APPROVED）、`[countryCode]`（按国家筛）。
- 唯一约束本身也是索引：`Booking.[teacherId, slotTime]`、`RefreshToken.tokenHash`、`Review.bookingId`。
- 原则：在「高频查询条件」和「需要保证唯一」的字段上建索引。

### Q9：为什么用 PostgreSQL 而不是 MongoDB？
**考点**：选型判断。
**要点**：
- 数据是强关系的（用户-外教-预约-评价层层关联），关系型天然合适。
- 需要数据库级的唯一约束、外键、事务来保证一致性（防重复预约就靠唯一约束）。
- 同时 Postgres 的 `JSON` 字段类型保留了灵活性（`learningGoals`、`Notification.payload` 用 JSON 存结构不固定的数据），不必为了灵活性放弃强一致。

### Q10：ORM 用 Prisma，它的事务怎么用的？
**考点**：事务理解。
**要点**：项目里用 `prisma.$transaction` 的地方，比如「外教设置可约时段」——要先删除旧的全部 WeeklySlot、再批量插入新的，这两步必须是原子的，否则中途失败会留下半套数据。教师列表查询也用事务同时拿「当前页数据」和「总数」，保证两者一致。

## 三、认证类

### Q11：access token 和 refresh token 的职责区别？
**考点**：双 token 方案理解。
**要点**：
- **access token**：短命（15min），每次请求都带，用来证明「我是谁」。短命是为了泄露后窗口小。
- **refresh token**：长命（30天），只在 access token 过期时用一次，用来「换一对新 token」。
- 分工本质：用一个短命的高频凭证 + 一个长命的低频凭证，兼顾安全和体验。

### Q12：token 存在哪里？为什么？
**考点**：前端安全。
**要点**：
- access token 存前端（内存 + localStorage），因为每个请求都要主动带在 `Authorization` 头里,JS 必须读得到。
- refresh token 存 `httpOnly` cookie，JS **读不到**——因为它价值高、又不需要 JS 主动操作（刷新接口靠浏览器自动带 cookie），放 httpOnly 能挡住 XSS 偷取。
- 诚实补充：access token 放 localStorage 仍有 XSS 风险，更严格的做法是只放内存。

### Q13：access token 过期了怎么自动续期、用户无感知？
**考点**：无感刷新实现。
**要点**：
- 前端 axios 响应拦截器捕获 401 → 自动调 `/auth/refresh` → 拿到新 token → 自动重发原失败请求。
- 两个关键细节：①**单飞刷新**——并发的多个 401 共用一次刷新请求，避免重复刷新；②**`_retry` 标记**——重试过的不再重试，防死循环。
- 对业务代码完全透明。
- （详见 [04 难点 2](./04-难点与解决方案.md#难点-2--jwt-双-token-认证与无感刷新)）

### Q14：JWT 怎么「登出 / 吊销」？
**考点**：JWT 的固有局限是否清楚。
**要点**：
- JWT 是无状态的，签发出去在有效期内本身无法「撤回」。
- 项目的处理：access token 有效期只有 15 分钟，能接受这个短窗口；登出时把 **refresh token 在数据库标记 `revokedAt`**，这样 access token 一过期就再也换不到新的了。
- 这就是「无状态 access + 有状态 refresh」的折中：既享受 JWT 的扩展性，又保留了吊销能力。

## 四、状态管理类（前端）

### Q15：React Query 和 Zustand 在你项目里各管什么？为什么这么分？
**考点**：状态分类的认知（前端重点必问）。
**要点**：
- **关键认知**：前端状态分「客户端状态」和「服务器状态」。服务器状态本质是服务器数据的缓存，有「会过期、要同步、可能并发请求」的特性。
- **React Query 管服务器状态**：教师列表、预约、通知——它提供缓存、请求去重、后台同步、失效重拉。
- **Zustand 只管客户端状态**：就一个 `useAuthStore`，管 user / token / 登录态。
- 为什么这么分：用大全局 store 管服务器数据，等于要自己手写一个半成品的缓存层（loading/error/失效/去重全得自己来）。React Query 把这些都做好了。
- （详见 [04 难点 3](./04-难点与解决方案.md#难点-3--前端状态管理架构的演进从全局-store-到-react-query)）

### Q16：React Query 的 staleTime 和 gcTime 是什么？
**考点**：是否真的理解所用工具。
**要点**：
- **staleTime**（项目设 30 秒）：数据「新鲜」的时长。新鲜期内组件再挂载/重渲染不会重新请求，直接用缓存。
- **gcTime**（项目设 5 分钟）：数据在没有任何组件使用后，在缓存里保留多久才被垃圾回收。
- 区别：staleTime 管「要不要重新请求」，gcTime 管「没人用了留多久」。

### Q17：一个写操作（比如创建预约）成功后，怎么让相关页面都刷新到最新？
**考点**：缓存失效策略。
**要点**：
- 用 `useMutation`，在 `onSuccess` 里 `queryClient.invalidateQueries`，**精确失效**相关的 queryKey（如 `['bookings']`、`['teacher', teacherId]`）。
- 被失效的 query 如果正在被页面使用，会自动重新拉取，所有用到它的组件一起刷新。
- 不用手动去各个组件通知——这正是用 React Query 替代全局 store 的核心收益。

## 五、实时通信类

### Q18：WebSocket 连接怎么鉴权？
**考点**：WebSocket 和 HTTP 鉴权的差异。
**要点**：
- WebSocket 是独立连接，不会自动带 `Authorization` 头，不能直接复用 HTTP 那套。
- 做法：前端在 socket **握手的 `auth` 字段**里带上 access token；后端 `handleConnection` 里 `jwt.verifyAsync` 验签，验不过直接 `disconnect`。
- 验签拿到 userId 后做后续的 room 绑定。

### Q19：怎么保证通知不会推送给错误的用户？
**考点**：定向投递机制。
**要点**：
- 后端在连接建立时 `client.join(`user:${userId}`)`，每个用户进入**自己专属的 room**。
- 推送时 `server.to(`user:${userId}`).emit(...)`，只投递到目标用户的 room。
- 从机制上保证定向投递，不会串台。

### Q20：实时推送来的通知，怎么和前端已有数据保持一致？
**考点**：实时数据 + 缓存一致性。
**要点**：
- 不直接把推送来的数据手动拼进前端 state（容易和服务器不一致）。
- 而是收到事件后 `invalidateQueries(['notifications'])`，让 React Query **重新拉取权威数据**。
- 后端也是「先写库再推送」，所以离线用户下次打开也能补看。
- （详见 [04 难点 6](./04-难点与解决方案.md#难点-6--实时通知websocket-鉴权与缓存一致性)）

## 六、AI / 后台任务类

### Q21：AI 匹配是怎么实现的？Gemini 调用失败了怎么办？
**考点**：第三方 AI 集成 + 健壮性。
**要点**：
- 取评分 Top 50 的已审核外教做粗筛（控制 prompt 体积和成本），连同学生学习目标拼成 prompt 发给 `gemini-2.0-flash-001`，要求返回固定 JSON（teacherIds + reasoning），并校验返回的 id 都合法。
- **降级**：没配 API key 或调用抛异常时，自动 fallback 到关键词启发式匹配（标签命中 +3 分 + 评分排序）。
- 所以 `/ai/match` 永远有结果，不会因为 AI 服务挂了而整个功能不可用。
- 还用 `@nestjs/throttler` 限流防滥用。

### Q22：课程提醒这种定时任务，为什么用 BullMQ 不用 cron？
**考点**：任务调度选型。
**要点**：
- cron 是「固定周期跑」，适合周期性任务；课程提醒是「每个预约各自的、未来某个精确时刻」触发的。
- 用 cron 实现得每分钟全表扫描找「谁到点了」，浪费且不精确。
- BullMQ：接受预约时直接 `queue.add(job, { delay })` 延时入队，到点 worker 自动消费。任务持久化在 Redis,服务重启不丢。
- 而「自动完成预约」那种本来就是周期性的，用 `setInterval` 的 sweep 循环就够了——不同任务用各自最合适的机制。
- （详见 [04 难点 7](./04-难点与解决方案.md#难点-7--后台定时任务为什么用-bullmq-而不是-cron)）

### Q23：你的后端依赖了 Redis、S3、Gemini、邮件服务，本地开发没有这些怎么办？
**考点**：优雅降级 / 故障隔离。
**要点**：
- 全部设计成**可选依赖**：构造函数检测配置，没配就把 client 设 null + 打 warning，调用处判断走降级分支。
- Gemini→启发式匹配；Redis→只跑 sweep 循环；S3→mock URL；邮件→打日志。
- 效果：`git clone` 不配任何 key 就能把后端跑起来，核心流程全可用；生产配齐 key 自动启用全部能力，代码不用改。
- 本质是「第三方服务不可用不传导成核心业务不可用」。
- （详见 [04 难点 5](./04-难点与解决方案.md#难点-5--可选外部服务的优雅降级)）

## 七、开放 / 反思类

### Q24：这个项目你觉得还有什么不足？如果继续做会怎么优化？
**考点**：自我反思能力（这题答得好是加分项，切忌说「没什么不足」）。
**要点**（挑 3-4 个讲，都来自 [04](./04-难点与解决方案.md) 的「延伸思考」）：
- **测试覆盖不足**：配了 Vitest 但用例少，核心的预约状态机、认证流程应该补单元测试和 E2E。
- **时区没完全做透**：外教按自己时区设置可约时段这块还没完整落地（`teacherTimezone` 参数预留了但没用），目前是按 UTC 处理。
- **错误处理可以更统一**：前端目前每个地方各自 try-catch，可以做统一的错误提示（toast）和 error boundary。
- **缺乏可观测性**：没有接日志聚合、APM、错误监控（如 Sentry），生产出问题不好排查。
- **加载体验**：部分页面只有「Loading...」文字，可以上骨架屏。
- **写操作的副作用可靠性**：创建预约和发通知不在一个事务里，可以用 outbox 模式。
- **CI 没有 CD**：构建通过后还得手动部署，可以补自动部署。

### Q25：如果用户量涨到 100 倍，哪里会先出问题？怎么扩展？
**考点**：系统设计 / 容量思维。
**要点**：
- **后端是无状态的**（JWT 无状态、session 不存内存），可以直接水平扩展加实例。
- **数据库会先成瓶颈**：读多的话加只读副本；教师列表已经做了分页；热点查询靠索引。
- **WebSocket 连接数**：单机连接数有上限，多实例时需要用 Redis adapter 让 Socket.io 跨实例广播。
- **后台任务**：BullMQ 本身支持多 worker 消费;sweep 循环要注意多实例下只让一个跑（用分布式锁或 repeatable job）。
- **静态资源**：前端是纯静态的,直接上 CDN。
- **AI 调用**：Gemini 有速率和成本限制，靠 Top 50 粗筛控制成本,可以加缓存（相同 goals 缓存结果）。

### Q26：这个项目你最有成就感 / 学到最多的是什么？
**考点**：表达和总结能力。
**要点**（选一个真实的展开，别泛泛而谈）：
- 比如：「最有收获的是想清楚了**前端状态该怎么分类管理**——以前都是一个大 store 全塞，做这个项目时踩了坑、重构成 React Query + Zustand 双轨后，才真正理解'服务器状态'和'客户端状态'是两回事。」
- 或者：「学到最多的是**健壮性设计**——一开始代码强依赖 Redis/S3/Gemini，换个环境就跑不起来；后来给每个外部依赖都加了降级路径，才体会到'故障隔离'不是口号，是要落到每一处的。」

## 附：答题通用心法

1. **先结论后展开**：先一句话给答案，再讲细节，别绕圈子。
2. **承认权衡**：每个选型都讲「换来什么、代价是什么」，体现你不是无脑用。
3. **诚实边界**：没做完的（如时区）就说没做完，把它转成「我知道完整方案是什么」的加分项。
4. **往代码上落**：能说出具体文件、具体约束、具体错误码，比空谈概念可信得多。
5. **留钩子**：回答时主动埋一个延伸点（「这里还可以…」），引导面试官往你熟悉的方向问。
