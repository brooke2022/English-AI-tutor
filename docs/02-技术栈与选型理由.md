## 2. 关键选型理由（逐项拆解）

### 2.1 后端框架：NestJS vs 原生 Express

**选 NestJS。**

- **问题**：原生 Express 灵活但没有约定，项目一大就容易变成「路由文件里塞业务逻辑」的意大利面，且难测试。
- **NestJS 带来的价值**：
  - **模块化**：按业务领域拆 Module（auth / bookings / teachers...），每个模块自带 Controller + Service，边界清晰。
  - **依赖注入（IoC）**：Service 之间通过构造函数注入，比如 `BookingsService` 注入了 `EmailService`、`NotificationsService`、`JobsService`，解耦且易于在测试里替换 mock。
  - **装饰器 + 守卫体系**：`@Public()` / `@Roles()` / `@CurrentUser()` 这些自定义装饰器让鉴权和权限控制变成声明式，全局守卫统一拦截。
  - **管道（Pipe）**：`ValidationPipe` + `class-validator` 把入参校验从业务逻辑里彻底剥离。
- **取舍**：NestJS 学习曲线更陡、有一定「框架魔法」，对小项目偏重。但这个项目有 13 个模块、3 种角色权限，NestJS 的结构化收益远大于成本。
- **底层说明**：NestJS 默认就是跑在 Express 之上的，所以并没有放弃 Express 生态，只是多了一层架构约束。

### 2.2 前端状态管理：React Query + Zustand「双轨制」

**用 React Query 管服务器状态，用 Zustand 只管认证状态。这是这个项目前端最值得讲的选型。**

- **问题**：项目早期前端是用 Zustand 写了 `useStore` 和 `useTeachersStore` 这种「大全局 store」，把教师列表、加载状态、错误状态全塞进去。结果是：每个页面要手写 `loading/error` 状态、手动触发刷新、数据容易过期、多个组件请求同一份数据会重复打接口。
- **关键认知**：前端状态其实分两类——
  - **客户端状态**：只属于浏览器、不需要和服务器同步的，比如「当前登录的用户是谁」「弹窗开没开」。
  - **服务器状态**：本质是服务器数据的一份缓存，有「会过期、要同步、可能并发请求」的特性，比如教师列表、预约列表、通知。
- **解决**：把这两类分开管。
  - **服务器状态全部交给 React Query**：自动缓存、自动去重（同一 queryKey 并发只发一次请求）、`staleTime` 控制新鲜度、mutation 成功后用 `invalidateQueries` 精确失效相关缓存。我把每个业务封装成 hook 放在 `src/hooks/`（`useTeachers` / `useBookings` / `useNotifications` ...），页面只调 hook。
  - **Zustand 只留 `useAuthStore`**：管 user / token / isAuthenticated 这些纯客户端态，并在应用启动时 `bootstrap()` 一次性恢复登录态。
- **结果**：删掉了 `useStore.ts` 和 `useTeachersStore.ts`，页面组件不再手写 loading/error，数据一致性和体验都变好。
- **取舍**：引入 React Query 增加了一个概念负担（queryKey 设计、缓存失效时机），但换来的是「不用自己造一个半成品的缓存层」。

> 详见 [04-难点与解决方案](./04-难点与解决方案.md) 难点 3。

### 2.3 ORM：Prisma vs 裸 SQL / TypeORM

**选 Prisma。**

- **类型安全**：`schema.prisma` 定义模型后，Prisma 生成完全类型化的 client。查询的返回值类型、`include` 进来的关联字段类型都是自动推导的，改了 schema 没改代码会编译报错。
- **schema 即文档**：一个 `schema.prisma` 文件就能看清所有表、字段、关系、索引、唯一约束。面试时直接拿这个文件讲数据模型很方便。
- **迁移管理**：`prisma migrate` 生成版本化的 SQL 迁移文件，生产环境用 `prisma migrate deploy` 自动应用（写在 Docker 启动命令里）。
- **vs TypeORM**：TypeORM 的类型推导没有 Prisma 彻底，装饰器式定义也比较啰嗦。
- **vs 裸 SQL**：裸 SQL 性能可控但开发效率低、易出注入漏洞、没有类型。
- **取舍**：Prisma 在非常复杂的查询上不如裸 SQL 灵活（可以用 `$queryRaw` 兜底），且多了一层 query engine。对这个项目的查询复杂度来说，开发效率和类型安全的收益更重要。

### 2.4 数据库：PostgreSQL

- **关系建模**：用户/外教/预约/评价之间是强关系数据，关系型数据库天然合适。
- **唯一约束防并发**：`Booking` 表上 `@@unique([teacherId, slotTime])`，从数据库层面杜绝同一外教同一时段被重复预约——这是解决并发竞态的关键（详见难点 1）。
- **JSON 字段的灵活性**：`StudentProfile.learningGoals`、`Notification.payload` 用 `Json` 类型，既享受关系型的强一致，又对「结构不固定的附加数据」保留弹性。
- **索引设计**：在高频查询字段上建了索引，如 `Booking` 的 `[studentId, status]` / `[teacherId, status]`、`TeacherProfile` 的 `[status]` / `[countryCode]`。

### 2.5 任务队列：Redis + BullMQ（为什么不用 cron）

**选 BullMQ 做「课前提醒」这类延时任务。**

- **问题**：外教接受预约后，需要在「课前 24 小时」和「课前 1 小时」给学生发提醒。这是**针对每个预约、在未来某个精确时间点**触发的任务。
- **为什么不用 cron**：cron 是「固定时间周期性跑」，要用它实现「每个预约各自的提醒时间」，就得每分钟扫全表、算哪些到点了——既浪费又不精确。
- **BullMQ 的做法**：接受预约时直接 `queue.add(jobData, { delay })`，把任务延时入队，到点了 worker 自动消费。任务持久化在 Redis 里，服务重启也不会丢。
- **同时保留了一个 sweep 兜底循环**：每分钟扫一次「已确认但已过课程时间 + 60 分钟宽限」的预约，自动标记为 COMPLETED。这个用定时循环就够了，因为它本来就是周期性的。
- **降级设计**：没配 `REDIS_URL` 时，BullMQ 不启动，但 sweep 循环照常跑——核心的「自动完成」不受影响，只是少了提前提醒。

> 详见难点 7。

### 2.6 实时通信：Socket.io vs 轮询

**选 Socket.io（WebSocket）。**

- **问题**：通知（预约被接受/拒绝、课程提醒）需要让用户**尽快看到**。
- **为什么不用轮询**：前端每隔几秒打一次 `/notifications`，绝大多数请求都是「没有新通知」的空响应，浪费带宽和服务器资源，且实时性受轮询间隔限制。
- **Socket.io 的做法**：客户端建立一条长连接，后端在事件发生时**主动推送**。后端用 `RealtimeGateway` 按 `user:{userId}` 把每个用户放进独立 room，推送时只发给目标用户的 room，不会串台。
- **为什么用 Socket.io 而不是裸 WebSocket**：Socket.io 自带断线重连、房间（room）抽象、握手鉴权钩子，省了很多基础设施代码。
- **取舍**：长连接对服务器有连接数压力，且部署时反向代理要正确转发 Upgrade 头。

> 详见难点 6。

### 2.7 认证：JWT + Refresh Token vs Session

**选 JWT access token + Refresh token 双 token 方案。**

- **无状态、易水平扩展**：JWT 自带签名，后端验签即可，不需要像 session 那样每次请求查 session 存储。多实例部署时不用共享 session。
- **双 token 的分工**：
  - **access token**：JWT，有效期短（15 分钟），放在前端内存 + localStorage，每次请求带在 `Authorization` 头里。
  - **refresh token**：随机 48 字节，有效期长（30 天），**SHA256 哈希后存数据库**，明文只通过 httpOnly cookie 下发——前端 JS 读不到，降低 XSS 风险。
- **刷新即轮换**：每次用 refresh token 换新 token 时，旧的立即作废、发一个新的，降低 refresh token 被盗用后的风险窗口。
- **vs 纯 Session**：Session 更易「立即吊销」，但有状态、扩展性差。这里用「access token 短有效期 + refresh token 可在库里标记 revoked」来折中。
- **取舍**：JWT 在有效期内无法即时吊销（access token 只有 15 分钟，可接受）。

> 详见难点 2。

### 2.8 文件上传：S3 预签名 URL vs 服务端代理

**选预签名 URL（客户端直传）。**

- **问题**：外教要上传头像、自我介绍视频（视频最大 200MB）。
- **为什么不让服务端代理**：如果文件先传到 NestJS、再由它转发到 S3，文件字节会**穿过后端**——占用后端带宽、内存，大文件还可能拖垮进程。
- **预签名 URL 的做法**：前端先调 `/uploads/presign` 拿一个有时效（5 分钟）的签名 URL，然后**直接 PUT 到 S3**，后端全程不碰文件字节。后端只做两件事：校验文件类型/大小、生成签名 URL。
- **降级**：没配 S3 时返回 mock URL，开发环境不阻塞。

### 2.9 AI 匹配：Google Gemini

- **产品价值**：把「学生用自然语言描述的学习目标」翻译成「最合适的 3 位外教 + 推荐理由」，降低选择成本——这是项目的差异化卖点。
- **实现**：取评分最高的 50 位已审核外教，连同学生目标拼成 prompt 发给 `gemini-2.0-flash-001`，要求返回固定 JSON 结构。
- **健壮性**：Gemini 没配 key 或调用失败时，自动降级为**关键词启发式匹配**（按标签命中 + 评分打分排序），保证 `/ai/match` 永远有结果。
- **防滥用**：用 `@nestjs/throttler` 限流。

> 详见难点 5。

### 2.10 构建工具：Vite vs CRA / Webpack

**选 Vite。**

- 开发环境基于原生 ESM + esbuild，**冷启动和 HMR 几乎瞬时**，相比 Webpack/CRA 体验差距明显。
- 生产构建用 Rollup，开箱即用的代码分割。
- 配置简洁，配 React 插件和 Tailwind 插件即可。

### 2.11 部署：Docker 多阶段构建 + Fly.io

- **多阶段构建**：
  - 前端：第一阶段 Node 镜像装依赖、`vite build` 出静态资源；第二阶段只用 Nginx 镜像托管产物——最终镜像不含 Node 和 node_modules，非常小。
  - 后端：第一阶段编译 TS、`prisma generate`、`npm prune` 删掉 devDependencies；第二阶段用精简 Node 镜像、非 root 用户运行，启动命令是 `prisma migrate deploy && node dist/...`，部署即自动迁移数据库。
- **为什么 Fly.io**：支持直接用 Dockerfile 部署、自带健康检查、可按区域就近部署、前后端可以各自作为独立 app。
- **CI**：GitHub Actions 在 push/PR 时跑前端 lint+build、后端 build+test，main 分支额外跑一次 Docker 构建 smoke test，确保镜像能正常构建。

## 3. 一句话选型总结表（速记）

| 选型 | 一句话理由 |
|------|-----------|
| NestJS | 模块化 + 依赖注入 + 守卫体系，适合多模块、多角色的中型项目 |
| React Query + Zustand | 区分服务器状态和客户端状态，前者要缓存/同步，后者只是本地态 |
| Prisma | 类型安全 + schema 即文档 + 版本化迁移 |
| PostgreSQL | 强关系数据 + 唯一约束防并发 + JSON 字段保留弹性 |
| BullMQ | 「未来精确时间点」的延时任务，cron 做不到 |
| Socket.io | 服务端主动推送，省掉无效轮询 |
| JWT 双 token | 无状态易扩展，短 access + 可吊销 refresh 折中安全性 |
| S3 预签名 URL | 大文件不穿过后端，省带宽和内存 |
| Gemini + 启发式降级 | AI 提升匹配体验，降级保证可用性 |
| Vite | 极快的冷启动和 HMR |
| Docker 多阶段 + Fly.io | 镜像小、部署即迁移、前后端独立伸缩 |
