# 04 · 难点与解决方案（STAR 结构）

> 这份文档帮你应对「你在这个项目里遇到过什么难点、是怎么解决的」。这是面试的**亮点输出环节**，每个难点都用 STAR 结构组织：**情境(S)** 当时什么背景 → **任务(T)** 要解决什么 → **行动(A)** 具体怎么做 → **结果(R)** 效果如何。
>
> 用法建议：挑 3-4 个你最熟、最能展开的，练到能脱稿讲。每个难点都标了对应代码文件，面试前对着代码再过一遍。
>
> ⚠️ 标了「**延伸思考**」的部分是项目里**尚未实现**的优化方向，讲的时候要说「这是我后续会做的」，别说成已完成。

## 难点 1 · 并发下的「重复预约」竞态

**对应代码**：`server/src/bookings/bookings.service.ts` 的 `create()`、`server/prisma/schema.prisma` 的 `Booking` 模型

### S 情境
预约系统的核心约束是：**同一个外教的同一个时间点，只能被预约一次**。但如果两个学生几乎同时点了同一个时段，会发生什么？

### T 任务
要保证在并发请求下，绝不会出现「同一外教同一时段」有两条预约记录。

### A 行动
我没有用「先查询有没有冲突，再插入」这种应用层判断——因为「查」和「插」之间有时间窗，两个并发请求可能都查到「没冲突」然后都插入，这是典型的 **check-then-act 竞态**。

我的做法是**把唯一性约束下沉到数据库**：
- 在 `Booking` 表上加复合唯一约束 `@@unique([teacherId, slotTime])`。
- 创建预约时直接 `INSERT`，让数据库来裁决。
- 用 `try/catch` 捕获 Prisma 的 `P2002`（唯一约束冲突）错误码，转成对用户友好的 `409 ConflictException`：「这个时段刚被预约了，请换一个时间」。

```ts
try {
  const booking = await this.prisma.booking.create({ data: {...} });
  // ... 发通知
} catch (e) {
  if (e instanceof Prisma.PrismaClientKnownRequestError && e.code === 'P2002') {
    throw new ConflictException('This slot has just been booked...');
  }
  throw e;
}
```

### R 结果
无论多少并发请求打到同一个时段，数据库的唯一约束保证**只有一条能成功**，其余的拿到明确的 409 提示。这个方案不需要加分布式锁、不需要事务串行化，性能好且绝对可靠。

### 延伸思考
- 当前 `create()` 里「插预约」和「发邮件 / 推通知」不在同一个事务里，如果通知失败预约仍然成功——可以引入「事务 + 事件表（outbox 模式）」让副作用更可靠。

## 难点 2 · JWT 双 token 认证与「无感刷新」

**对应代码**：`server/src/auth/auth.service.ts`、`server/src/auth/auth.controller.ts`、`src/lib/api.ts`

### S 情境
用 JWT 做认证时有个经典矛盾：access token 有效期设长了不安全（泄露后长期可用），设短了用户体验差（频繁掉登录）。

### T 任务
既要 access token 短命保证安全，又要用户感知不到「token 过期」——不能让用户隔 15 分钟就被踢去重新登录。

### A 行动
我用了 **access token + refresh token 双 token 方案**，并在前端做了无感刷新：

**后端（`auth.service.ts`）：**
- access token：JWT，15 分钟有效，payload 只放 `{ sub, email, role }`。
- refresh token：48 字节随机串，30 天有效。**关键是它不明文存库**——存的是 `SHA256(refreshToken)`，且只通过 `httpOnly + secure + sameSite=lax` 的 cookie 下发，前端 JS 读不到。
- **刷新即轮换**：每次 `/auth/refresh`，先把旧 refresh token 标记 `revokedAt`，再发一对全新的。万一旧 token 泄露，用一次就失效。

**前端（`src/lib/api.ts`）axios 双拦截器：**
- 请求拦截器：自动把 access token 加到 `Authorization` 头。
- 响应拦截器：捕获 401 → 自动调 `/auth/refresh`（cookie 会自动带上 refresh token）→ 拿到新 token 后**自动重发原来那个失败的请求**。
- 两个细节防坑：
  - **单飞刷新**：用一个共享的 `refreshing` Promise，并发的多个 401 只触发一次刷新，刷完一起重试。
  - **`_retry` 标记**：已经重试过的请求不再二次重试，防死循环。

### R 结果
用户完全感知不到 token 过期——业务代码里调接口该怎么写还怎么写，token 刷新对上层完全透明。安全性上：access token 窗口短、refresh token 哈希入库且 XSS 偷不到、还能轮换吊销。

### 延伸思考
- access token 目前存在 localStorage，相比纯内存更易受 XSS 影响——可以改成只存内存、靠刷新接口恢复。
- refresh token 轮换可以再加「复用检测」：如果一个已 revoked 的 token 又被使用，说明可能被盗，立即把该用户所有 refresh token 全部吊销。

## 难点 3 · 前端状态管理架构的演进（从全局 store 到 React Query）

**对应代码**：删除的 `src/store/useStore.ts`、`src/store/useTeachersStore.ts`；新增的 `src/hooks/*`、`src/lib/queryClient.ts`

### S 情境
项目早期，前端所有数据都塞在 Zustand 的大全局 store 里（`useStore`、`useTeachersStore`），包括教师列表、加载状态、错误状态。随着页面变多，问题暴露出来：每个页面要手写 `loading/error`、数据拿了不知道什么时候过期、多个组件请求同一份数据会重复打接口、改完数据要手动通知各处刷新。

### T 任务
重构前端的数据层，解决数据一致性差、样板代码多、缓存缺失的问题。

### A 行动
核心是想清楚一件事：**前端状态分两类**——
- **客户端状态**：纯本地的，比如「当前登录用户」「弹窗开没开」。
- **服务器状态**：本质是服务器数据的缓存，有「会过期、要同步、可能并发」的特性，比如教师列表、预约、通知。

这两类该用不同工具管：
- **服务器状态全部迁到 TanStack React Query**：
  - 在 `lib/queryClient.ts` 统一配置：`staleTime` 30 秒、`gcTime` 5 分钟、401/403/404 不重试、其余错误重试 2 次。
  - 每个业务封装成自定义 hook 放 `src/hooks/`：`useTeachers` / `useBookings` / `useNotifications` / `useAdmin` / `useAiMatch`。页面组件只调 hook，拿到 `{ data, isLoading, error }`，不再手写状态。
  - 写操作用 `useMutation`，成功后 `invalidateQueries` 精确失效相关缓存，自动触发重新拉取。
- **Zustand 只保留 `useAuthStore`**：管 user / token / isAuthenticated，并在应用启动时 `bootstrap()` 恢复登录态。
- 最终**删掉了 `useStore.ts` 和 `useTeachersStore.ts`**。

### R 结果
- 页面组件不再有样板化的 loading/error 代码。
- React Query 自动做请求去重（同 queryKey 并发只发一次）、后台同步、缓存失效。
- 数据一致性问题消失：一次 mutation 后所有相关页面自动刷新到最新。
- 职责清晰：`lib/` 是基础设施、`hooks/` 是数据获取、`store/` 只剩认证。

### 延伸思考
- 可以给关键 mutation 加**乐观更新（optimistic update）**，比如「标记通知已读」立即更新 UI、失败再回滚，体验更跟手。

## 难点 4 · 跨时区的时间一致性

**对应代码**：`server/src/teachers/teachers.service.ts` 的 `computeAvailableSlots`、`server/prisma/schema.prisma` 的 `WeeklySlot` / `Booking`、`src/utils/time.ts`

### S 情境
这是个**全球外教**平台，外教和学生大概率不在同一个时区。时间这块如果设计不好，会出现「外教以为约的是自己时间的下午 2 点，学生看到的是别的时间」这种灾难。

### T 任务
要让时间在「外教设置 → 存储 → 学生查看」整条链路里保持一致、不出错。

### A 行动
我定了一个原则：**存储和计算层永远只用 UTC，时区转换只发生在最外层的展示环节**。
- **外教的可约时段**用「星期几 + 小时」的周模板（`WeeklySlot`），按 UTC 理解。
- **预约的具体时间** `Booking.slotTime` 是 UTC 的 `DateTime`。
- 后端 `computeAvailableSlots` 把周模板展开成未来 4 周的具体时间点时，全程用 `getUTCDay()` / `setUTCHours()` 等 UTC API，输出 ISO UTC 字符串。
- 前端 `src/utils/time.ts` 用 `Intl.DateTimeFormat` 把 UTC 字符串转成**浏览器本地时区**显示给用户。

### R 结果
整条链路只有「展示」这一处做时区转换，存储和计算永远是 UTC，从架构上消除了「时区在中间层被转来转去」导致的不一致。

### 延伸思考（这点要诚实说）
- 当前实现里，外教设置的「周模板小时」是直接按 UTC 存的，`computeAvailableSlots` 也预留了 `teacherTimezone` 参数但还没真正使用——所以严格来说「按外教所在时区设置可约时段」还没完整落地。
- 完整方案应该是：外教按**自己时区**设置 → 后端结合 `teacherTimezone` 转成 UTC 存 → 学生端再转成**学生时区**展示。这是我清楚知道、但还没做完的一块，可以作为改进点来讲。

## 难点 5 · 可选外部服务的「优雅降级」

**对应代码**：`server/src/ai/ai.service.ts`、`server/src/jobs/jobs.service.ts`、`server/src/uploads/uploads.service.ts`、`server/src/notifications/email.service.ts`

### S 情境
后端依赖了好几个第三方服务：Google Gemini（AI 匹配）、Redis（任务队列）、AWS S3（文件存储）、Resend（邮件）。但这些服务在本地开发、或者还没申请到 key 的时候是没有的。如果代码强依赖它们，「没配 key 就跑不起来」会非常影响开发和演示。

### T 任务
让这些外部依赖**全部可选**：配了就用真实服务，没配就降级，但**核心业务流程任何时候都不能被它们卡住**。

### A 行动
我给每个外部服务都设计了降级路径：

| 服务 | 配置缺失 / 失败时的降级 |
|------|------------------------|
| **Gemini AI** | 降级为关键词启发式匹配（标签命中 +3 分 + 评分排序）；即使配了 key，调用抛异常也会自动 fallback 到启发式 |
| **Redis / BullMQ** | 不启动任务队列，但「自动完成预约」的 sweep 循环照常跑——只是少了「课前提前提醒」 |
| **AWS S3** | 返回 mock URL，上传流程在开发环境不阻塞 |
| **Resend 邮件** | 降级为打印到控制台日志，站内通知不受影响 |

实现上统一用一个模式：构造函数里检测配置 → 配了就初始化 client、没配就把 client 设为 null + 打 warning 日志 → 调用处判断 client 是否为 null 走不同分支。

### R 结果
- `git clone` 下来，**不配任何第三方 key 就能把整个后端跑起来**，核心的注册/预约/通知流程全部可用。
- 生产环境配齐 key 后自动启用全部能力，代码无需改动。
- 这本质是「关注点隔离」+「故障隔离」：第三方服务的不可用，不会传导成核心业务的不可用。

### 延伸思考
- 可以给降级状态加一个监控指标 / 健康检查项，让运维知道「现在正跑在降级模式」。

## 难点 6 · 实时通知：WebSocket 鉴权与缓存一致性

**对应代码**：`server/src/realtime/realtime.gateway.ts`、`server/src/notifications/notifications.service.ts`、`src/hooks/useNotifications.ts`

### S 情境
预约被接受/拒绝、课程提醒这些事件，需要让用户尽快看到。用轮询的话大量请求是空响应、浪费且不实时。

### T 任务
做一套实时通知，要解决两个关键问题：①WebSocket 连接怎么鉴权、怎么保证通知不发错人；②实时推送来的数据怎么和前端已有的缓存保持一致。

### A 行动
**① 鉴权与定向投递**：
- WebSocket 是独立于 HTTP 的连接，不会自动带 `Authorization` 头。我让前端在 socket **握手的 `auth` 字段里带上 access token**。
- 后端 `RealtimeGateway.handleConnection` 里 `jwt.verifyAsync` 验签，验不过直接 `disconnect`。
- 验签拿到 `userId` 后，`client.join(`user:${userId}`)` 把每个连接放进**自己专属的 room**。
- 推送时 `server.to(`user:${userId}`).emit(...)`，只发给目标用户的 room——从机制上杜绝「通知发错人」。

**② 写库 + 推送双管**：
- `NotificationsService.push()` 先往 `Notification` 表插一行（持久化），再通过网关推送。
- 在线用户实时收到；离线用户下次打开调 `/notifications` 也能补看历史。

**③ 推送不直接改缓存，而是触发失效**：
- 前端 `useRealtimeNotifications` 收到 `notification` 事件后，**不是手动把这条通知拼进本地 state**，而是 `queryClient.invalidateQueries(['notifications'])` 和 `(['bookings'])`。
- 让 React Query 重新拉取权威数据，避免「手动拼的本地状态和服务器不一致」。

### R 结果
通知近实时到达；per-user room 保证定向投递不串台；「写库 + 推送 + 失效重拉」三步保证了在线/离线、实时性/一致性都兼顾。

### 延伸思考
- 当前 WebSocket 用 access token 鉴权，token 过期后这条连接会失效——可以加「token 快过期时前端主动重连」或服务端定期校验。

## 难点 7 · 后台定时任务：为什么用 BullMQ 而不是 cron

**对应代码**：`server/src/jobs/jobs.service.ts`

### S 情境
外教接受预约后，要在**课前 24 小时**和**课前 1 小时**给学生发提醒。

### T 任务
实现「针对每一个预约、在未来某个精确时间点」触发的任务。

### A 行动
- **为什么 cron 不合适**：cron 是固定周期跑。要用它实现「每个预约各自的提醒时刻」，就得每分钟扫全表算「谁到点了」，既浪费又不精确。
- **用 BullMQ（Redis 为后端）**：外教接受预约的那一刻，直接 `reminderQueue.add(jobData, { delay })` 把任务**延时入队**——delay 算成「距离提醒时刻还有多久」。到点了 worker 自动消费，发邮件 + 推站内通知。任务持久化在 Redis 里，服务重启不丢。
- **同时保留一个 sweep 兜底循环**：每分钟扫一次「已确认 + 课程时间已过 60 分钟宽限」的预约，批量标记 COMPLETED。这种「周期性」的活儿用 `setInterval` 就够了，不需要队列。
- **降级**（呼应难点 5）：没配 Redis 时 BullMQ 不启动，但 sweep 循环照常跑。

### R 结果
- 提醒任务精确、不浪费——不需要每分钟全表扫描去找「该提醒谁」。
- 任务持久化，服务重启不丢。
- 「自动完成」和「定时提醒」用了各自最合适的机制（循环 vs 队列），且 Redis 不可用时核心的自动完成不受影响。

### 延伸思考
- 取消预约时目前没有取消已调度的提醒任务——可以在 cancel 时按 jobId 把对应的延时任务移除。

## 难点 8 · 预约状态机的「自动流转」

**对应代码**：`server/src/jobs/jobs.service.ts` 的 `sweepCompleted()`

### S 情境
课上完之后，预约状态该从 CONFIRMED 变成 COMPLETED，学生才能评价。但「课上完了」这个事件**没有人会主动来点一个按钮**——总不能指望外教或学生每次都手动「确认完课」。

### T 任务
让 CONFIRMED → COMPLETED 这个状态转换**自动发生**，不依赖任何人操作。

### A 行动
在 `JobsService` 里跑一个 sweep 循环：
- 每 60 秒执行一次。
- 查所有「状态 = CONFIRMED 且 slotTime 已经过去超过 60 分钟」的预约（60 分钟是给课程时长留的宽限期）。
- 用 `updateMany` **批量**把它们标记成 COMPLETED。

```ts
const cutoff = new Date(Date.now() - 60 * 60 * 1000); // 60 分钟宽限
await this.prisma.booking.updateMany({
  where: { status: CONFIRMED, slotTime: { lt: cutoff } },
  data: { status: COMPLETED },
});
```

### R 结果
预约状态机能自己「往前走」，课上完后学生总能拿到评价入口，整个业务闭环不依赖人工干预。用 `updateMany` 批量处理也避免了逐条更新的开销。

### 延伸思考
- 多实例部署时，多个实例的 sweep 循环会同时跑——虽然 `updateMany` 是幂等的（`where` 条件带了 `status: CONFIRMED`，重复跑也只会命中一次），不会出错，但可以用分布式锁或把 sweep 也放进 BullMQ 的 repeatable job 让它只在一个实例跑。

## 难点 9 · 前后端分离的生产部署

**对应代码**：`Dockerfile`、`server/Dockerfile`、`nginx.conf`、`fly.toml`、`server/fly.toml`、`.github/workflows/ci.yml`

### S 情境
项目是前后端分离的，要把它真正部署上线，而不是只能在本地 `npm run dev`。

### T 任务
让前后端都能容器化、可复现地部署，且部署流程尽量自动化。

### A 行动
**Docker 多阶段构建**，前后端各一套：
- **前端**：第一阶段用 Node 镜像装依赖、`vite build` 出静态资源；第二阶段**只用 Nginx 镜像**托管产物。最终镜像不含 Node 和 node_modules，非常小。Nginx 还配了 SPA 路由回退（`try_files ... /index.html`）、Gzip、对 hash 资源做长缓存、对 `index.html` 不缓存。
- **后端**：第一阶段编译 TS、`prisma generate`、`npm prune` 删 devDependencies；第二阶段用精简 Node 镜像、**非 root 用户**运行，启动命令是 `prisma migrate deploy && node dist/...`——**部署即自动迁移数据库**。

**部署到 Fly.io**：前后端各作为一个独立 app（tutor-web / tutor-api），各自配健康检查端点，可以独立伸缩。

**CI（GitHub Actions）**：push / PR 时并行跑「前端 lint + build」「后端 build + test」，main 分支额外跑一次 Docker 构建 smoke test，确保镜像始终可构建。

### R 结果
- 一条命令就能复现构建，不存在「在我机器上能跑」。
- 部署自动跑数据库迁移，不会忘。
- CI 在合并前就拦住构建失败。
- 前后端解耦部署，可以各自独立更新和伸缩。

### 延伸思考
- 目前 CI 只做了构建 smoke test，没有自动部署——可以加一个「main 分支构建通过后自动 deploy 到 Fly.io」的 CD 步骤。
- 多副本部署时，多个实例同时启动会都跑 `prisma migrate deploy`，可能产生迁移锁竞争——生产严谨做法是把迁移拆成独立的一次性 job。
