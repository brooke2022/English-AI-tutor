# 03 · 系统架构与数据流

> 这份文档帮你应对「画一下流程图 / 讲一下某个功能怎么实现的」。技术面里，面试官最爱顺着一个功能问「数据怎么流的」。下面每张图都标了对应的真实代码文件，建议对着代码各跑一遍。

## 1. 系统分层架构

### 1.1 后端模块化分层

每个业务模块内部都是清晰的三层：

```
HTTP 请求
   │
   ▼
┌─────────────┐   入参校验(ValidationPipe + class-validator DTO)
│ Controller  │   鉴权(JwtAuthGuard) + 权限(RolesGuard)
│  路由 + 装饰器 │   只负责「接收请求、调 Service、返回响应」
└──────┬──────┘
       ▼
┌─────────────┐   全部业务逻辑都在这里
│  Service    │   通过构造函数注入其它 Service（依赖注入）
│             │   例：BookingsService 注入 EmailService /
│             │        NotificationsService / JobsService
└──────┬──────┘
       ▼
┌─────────────┐
│ PrismaService│  类型安全的数据库访问
└──────┬──────┘
       ▼
   PostgreSQL
```

### 1.2 全局横切关注点（`server/src/main.ts` + `app.module.ts`）

请求进入任何 Controller 之前，会先经过这些全局处理：

| 层 | 作用 | 配置位置 |
|----|------|---------|
| Helmet | 设置安全响应头 | `main.ts` |
| CORS | 限定允许的来源 + 允许携带 cookie | `main.ts` |
| cookie-parser | 解析 refresh token cookie | `main.ts` |
| ValidationPipe | 全局入参校验，`whitelist` 剥离多余字段、`forbidNonWhitelisted` 拒绝未知字段 | `main.ts` |
| ThrottlerGuard | 全局限流：60 秒内每 IP 100 次请求 | `app.module.ts` |
| JwtAuthGuard | 默认所有端点都要鉴权，除非标了 `@Public()` | `auth` 模块全局守卫 |
| RolesGuard | 校验 `@Roles()` 声明的角色 | `auth` 模块全局守卫 |

## 2. 认证流程（注册 / 登录 / 无感刷新 / 登出）

涉及文件：`server/src/auth/auth.service.ts`、`auth.controller.ts`、`src/lib/api.ts`、`src/store/useAuthStore.ts`

### 2.1 注册 / 登录：签发双 token

```
前端                          后端 (auth.service.ts)
 │                              │
 │ POST /auth/register|login    │
 │─────────────────────────────▶│
 │                              │ 1. bcrypt 校验/哈希密码(12 轮)
 │                              │ 2. 签发 access token (JWT, 15min)
 │                              │    payload: { sub, email, role }
 │                              │ 3. 生成 refresh token
 │                              │    = crypto.randomBytes(48)
 │                              │ 4. SHA256(refreshToken) 存入
 │                              │    RefreshToken 表(明文不入库)
 │                              │
 │   200 { accessToken, user }  │
 │   + Set-Cookie: refresh_token│  ◀── httpOnly + secure(prod)
 │◀─────────────────────────────│       + sameSite=lax
 │                              │       + path=/api/v1/auth
 │ 5. accessToken 存内存 +       │
 │    localStorage              │
 │ 6. refresh_token 在 httpOnly │
 │    cookie 里，JS 读不到       │
```

**关键设计点（面试可主动说）**：
- access token 短命（15min），就算泄露窗口也小。
- refresh token 长命（30天）但**哈希入库** + **只走 httpOnly cookie**：数据库被拖库也拿不到可用 token；前端 XSS 也偷不到 cookie。
- cookie 的 `path` 限定为 `/api/v1/auth`，refresh token 只在认证相关请求里才会被带上。

### 2.2 无感刷新（前端 axios 双拦截器，`src/lib/api.ts`）

```
任意业务请求                           
 │                                    
 │ 请求拦截器：自动注入                  
 │ Authorization: Bearer <accessToken> 
 ▼                                    
后端  ──── access token 过期 ────▶ 401
 │                                    
 ▼ 响应拦截器捕获 401
 │  条件: 状态=401 且 不是 /auth/ 接口 且 该请求没重试过
 │
 │  ┌──────────────────────────────────────┐
 │  │ 单飞(single-flight)刷新:               │
 │  │ refreshing ??= performRefresh()       │  ← 并发的多个 401
 │  │ POST /auth/refresh (带 httpOnly cookie)│    只会触发一次刷新
 │  └──────────────┬───────────────────────┘
 │                 ▼
 │        刷新成功 → 拿到新 accessToken
 │                 │  → 更新内存 + localStorage
 │                 │  → 用新 token 重发原请求(标记 _retry 防死循环)
 │                 │
 │        刷新失败 → 清空 token，原 401 抛给业务层
```

**关键设计点**：
- **单飞刷新**：用一个模块级的 `refreshing` Promise，并发的多个 401 共用同一次刷新请求，刷完一起重试，避免「N 个请求触发 N 次刷新」。
- **`_retry` 标记**：重试过的请求不再二次重试，防止死循环。
- **排除 `/auth/` 接口**：登录、刷新本身返回 401 不该触发刷新逻辑。
- 对业务代码完全透明——页面里调 `useTeachers()` 根本不知道中间发生过一次 token 刷新。

> 详见 [04-难点与解决方案](./04-难点与解决方案.md) 难点 2。

### 2.3 应用启动时恢复登录态（`useAuthStore.bootstrap()`）

```
页面刷新 / 首次打开
 │
 ▼ useAuthStore 模块加载时自动调 bootstrap()
 │
 │ localStorage 里有 accessToken 吗?
 │   ├─ 没有 → 保持未登录
 │   └─ 有   → GET /auth/me
 │              ├─ 成功 → 恢复 user + isAuthenticated
 │              └─ 失败(且刷新也失败) → 清空，回到未登录
```

### 2.4 前端路由权限（`src/components/ProtectedRoute.tsx`）

```
访问 /teacher/dashboard
 │
 ▼ <ProtectedRoute role="teacher">
 │   未登录?            → 重定向 /login?redirect=原路径
 │   登录了但角色不符?   → 重定向到该角色自己的首页
 │   通过               → 渲染页面
```

## 3. 预约状态机（核心业务）

涉及文件：`server/src/bookings/bookings.service.ts`、`server/src/jobs/jobs.service.ts`

### 3.1 状态流转图

```
                  学生发起预约
                  POST /bookings
                       │
                       ▼
                 ┌───────────┐
                 │  PENDING  │ 待外教处理
                 └─────┬─────┘
            ┌──────────┼──────────────┐
   外教接受 │          │ 外教拒绝       │ 学生/外教取消
   +meetingUrl         │ +rejectionReason
            ▼          ▼              ▼
      ┌───────────┐  ┌───────────────────┐
      │ CONFIRMED │  │     CANCELLED     │ (终态)
      └─────┬─────┘  └───────────────────┘
            │   ▲
   学生/外教 │   │ 课程时间 + 60min 宽限期后
   可取消    │   │ 后台 sweep 自动流转
            │   │
            │   ▼
            │ ┌───────────┐
            └▶│ COMPLETED │ (终态) → 解锁学生评价入口
              └───────────┘
```

### 3.2 每个状态转换的「触发方 + 校验 + 副作用」

| 转换 | 触发方 | 前置校验 | 副作用 |
|------|--------|---------|--------|
| → PENDING（创建） | 学生 | slotTime 必须在未来；外教存在且已审核；不能预约自己 | 写 Booking 行（命中 `@@unique` 则抛 ConflictException）；发邮件给外教；推送 `BOOKING_CREATED` 通知 |
| PENDING → CONFIRMED | 外教 | 必须是该外教本人；当前必须是 PENDING | 写入 meetingUrl；**调度课前 24h / 1h 提醒任务**；发邮件 + 推送 `BOOKING_CONFIRMED` 给学生 |
| PENDING → CANCELLED（拒绝） | 外教 | 同上 | 写入 rejectionReason；发邮件 + 推送 `BOOKING_CANCELLED` 给学生 |
| PENDING/CONFIRMED → CANCELLED（取消） | 学生 / 外教 / 管理员 | 必须是预约相关方；只有 PENDING/CONFIRMED 可取消 | 发邮件 + 推送 `BOOKING_CANCELLED` 给「另一方」 |
| CONFIRMED → COMPLETED | 系统（后台 sweep） | slotTime 已过去 60 分钟以上 | 批量 `updateMany` 标记完成 |

**关键设计点**：
- 状态机的每个转换都做了**前置状态校验**（比如不能 accept 一个已 CANCELLED 的预约），非法转换直接抛 `BadRequestException`。
- 「自动完成」不依赖任何人操作，由后台循环兜底，保证课上完后学生总能评价。

> 详见难点 1（防重复预约）和难点 8（自动完成）。

## 4. 实时通知数据流

涉及文件：`server/src/notifications/notifications.service.ts`、`server/src/realtime/realtime.gateway.ts`、`src/lib/socket.ts`、`src/hooks/useNotifications.ts`、`src/components/NotificationBell.tsx`

```
① 业务事件发生（如外教接受预约）
   BookingsService 调 notifications.push(userId, type, payload)
                       │
                       ▼
② NotificationsService.push():
   - 往 Notification 表插一行（持久化，离线也能补看）
   - 调 realtime.emitToUser(userId, 'notification', {...})
                       │
                       ▼
③ RealtimeGateway.emitToUser():
   server.to(`user:${userId}`).emit('notification', payload)
   只发给目标用户的 room，不会串台
                       │
            WebSocket (/ws) 推送
                       ▼
④ 前端 socket 收到 'notification' 事件
   useRealtimeNotifications hook 里的监听器触发:
   - queryClient.invalidateQueries(['notifications'])
   - queryClient.invalidateQueries(['bookings'])
                       │
                       ▼
⑤ React Query 缓存失效 → 自动重新拉取 /notifications
                       │
                       ▼
⑥ NotificationBell 组件重新渲染，未读数 +1
```

**WebSocket 连接是怎么鉴权的**（`realtime.gateway.ts` 的 `handleConnection`）：
```
前端 connectSocket() 时把 accessToken 放进 socket 握手的 auth 字段
   │
   ▼
后端 handleConnection:
   - 从 handshake.auth.token 取 token
   - 没有 token → 立即 disconnect
   - jwt.verifyAsync 验签 → 拿到 userId
   - client.join(`user:${userId}`)  ← 加入自己的专属 room
   - 验签失败 → disconnect
```

**关键设计点**：
- **「写库 + 推送」双管**：通知先落库再推送。在线用户实时收到；离线用户下次打开调 `/notifications` 也能补看历史。
- **per-user room 隔离**：每个用户一个 room，推送时精确投递，从根本上避免「通知发错人」。
- **推送不直接改前端缓存，而是 `invalidateQueries`**：让 React Query 重新拉取，保证拿到的是完整、权威的数据，而不是手动拼一个可能不一致的本地状态。

> 详见难点 6。

## 5. AI 教师匹配流程

涉及文件：`server/src/ai/ai.service.ts`、`src/hooks/useAiMatch.ts`、`src/pages/AIMatch.tsx`

```
学生在 /ai-match 输入学习目标
   │ POST /ai/match { goals, level? }
   ▼
AiService.match():
   │
   ① 查 DB：取 status=APPROVED 的外教，按 (ratingAvg desc, reviewCount desc)
   │   排序，取 Top 50
   │
   ② 没有任何外教? → 直接返回空结果
   │
   ③ 配了 GEMINI_API_KEY?
   │   ├─ 是 → geminiMatch():
   │   │        把 50 位外教精简成 JSON(id/name/country/tags/intro/rating)
   │   │        + 学生目标 拼成 prompt
   │   │        调 gemini-2.0-flash-001
   │   │        从返回文本里截取 JSON，校验 teacherIds 都是合法 id
   │   │        返回 { teacherIds(≤3), reasoning }
   │   │        └─ 调用抛异常? → 自动 fallback 到 ④
   │   │
   │   └─ 否 / 失败 → ④ heuristicMatch():
   │            关键词启发式：标签命中学生目标 +3 分，
   │            叠加 ratingAvg 分，排序取 Top 3
   ▼
返回 { teacherIds, reasoning }
   │
   ▼
前端用 teacherIds 从 useTeachers() 已有缓存里取出完整外教信息，
配 motion 动画展示推荐结果 + 推荐理由
```

**关键设计点**：先用「评分 Top 50」做粗筛（控制 prompt 体积和成本），再交给 AI 精排。AI 不可用时无缝降级，`/ai/match` 永远有结果。

> 详见难点 5。

## 6. 外教可约时段的计算

涉及文件：`server/src/teachers/teachers.service.ts` 的 `computeAvailableSlots`

外教设置的是**每周重复的模板**（`WeeklySlot`：星期几 + 几点），学生看到的需要是**未来具体的可约时间点**。两者之间的转换：

```
外教的 WeeklySlot 模板（如：周一 14:00、周三 20:00，UTC）
   │
   ① 从「今天」起，往后展开 4 周
   │   逐天检查：这天的星期几 == 模板里某条的 dayOfWeek 吗?
   │   是 → 生成一个具体 UTC 时间点（该日期 + 模板的 hour）
   │   只保留「在未来 且 在 4 周内」的
   │   ⇒ 得到候选时间点列表 candidates[]
   │
   ② 查这个外教在这些候选时间点上、状态为
   │   PENDING 或 CONFIRMED 的已有预约
   │   ⇒ 得到 bookedSet
   │
   ③ candidates 减去 bookedSet
   ▼
返回学生可见的「可预约时间点」列表（ISO UTC 字符串）
```

**关键设计点**：
- 所有时间在数据库和计算里**统一用 UTC**，避免时区混乱。
- 「模板 + 展开」的设计让外教只需配一次每周规律，不用逐周手填。
- 设置可约时段时用事务「先全删再批量插」，保证模板是一个原子的整体替换。

> 时区相关的取舍详见难点 4。

## 7. 文件上传流程（S3 预签名 URL）

涉及文件：`server/src/uploads/uploads.service.ts`、`src/hooks/useUpload.ts`

```
外教选择头像/视频文件
   │ ① POST /uploads/presign { kind, filename, contentType, size }
   ▼
UploadsService.presign():
   - 按 kind 校验大小上限（头像 5MB / 视频 200MB）和 MIME 类型
   - 生成随机对象 key：avatars/{userId}/{随机}.{ext}
   - 配了 S3? 
       是 → getSignedUrl() 生成 5 分钟有效的 PUT 预签名 URL
       否 → 返回 mock URL（开发环境不阻塞）
   │ 返回 { uploadUrl, publicUrl, key }
   ▼
② 前端拿 uploadUrl，直接 PUT 文件到 S3（不经过后端）
   │
   ▼
③ 上传成功后，用 publicUrl 作为头像/视频地址保存到用户资料
```

**关键设计点**：后端只做「鉴权 + 校验 + 签名」，文件字节全程不经过后端，省带宽省内存。

## 8. 一张图记住所有数据流的共性

```
所有「写操作」都遵循同一个模式：

  Controller(校验+鉴权)
        → Service(业务校验 + 状态机校验)
            → Prisma 写库（唯一约束兜底并发）
                → 触发副作用（邮件 / 站内通知 / 调度任务）
                    → 通知通过 WebSocket 实时推送
                        → 前端 invalidateQueries 重新拉取
```

把这个「主干」记牢，面试时无论问哪个具体功能，都能往这个框架上套。
